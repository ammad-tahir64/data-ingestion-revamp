# Phase 1: DeviceSummaries Index Optimization

## Purpose

The `DeviceSummaries` table has accumulated **23.6 million rows (566 GB)** and currently carries **7 indexes** — 4 of which are severely bloated, with key column counts ranging from 20 to 35 columns. These oversized indexes are near-complete copies of the table itself.

Every device ping (49K devices × 65–157 pings/day) triggers a write to **all 7 indexes**, causing excessive DTU consumption on the Azure SQL database. This Phase 1 maintenance drops the 4 bloated indexes and replaces them with 1 purpose-built index that covers the actual query patterns.

**No data is changed. No schema columns are altered. Only index definitions are modified.**

| | Before | After |
|---|---|---|
| Total indexes | 7 | 4 |
| Largest index (columns in key) | 35 | 5 |
| Estimated total index storage | ~1.5–2 TB | < 50 GB |
| Write ops per ping | 1 insert + 7 index updates | 1 insert + 4 index updates |
| Estimated DTU reduction | — | 40–60% on write-heavy workload |

| Index Name | Type | Reason Kept |

| `IX_DeviceSummaries_CompanyId_IMEI_TimeStamp` | Non-clustered | Serves legitimate multi-tenant query patterns |

### Indexes Dropped

| Index Name | Columns | Reason Dropped |
|---|---|---|
| `IX_DeviceSummaries_DeviceId` | 26 | Near-full table copy — not a selective index |
| `nci_msft_DeviceSummaries_BC08984D5BACAEE1C87569072C4FECC0` | 35 | Worst offender — auto-generated by Azure SQL advisor, ~566 GB duplicate |
| `nci_wi_DeviceSummaries_DDD72ECE7AF305F90C5D1276C40FA2C6` | 20 | Heavy overlap with other indexes, auto-generated |

### New Index Created

| Index Name | Key Columns | Included Columns |
|---|---|---|
| `IX_DeviceSummaries_IMEI_TimeStamp_IsMove` | `IMEI`, `TimeStamp DESC` | `IsMove`, `CompanyId`, `AssetId` |

## Execution Order

Run scripts **in sequence**. Do not skip steps or run them in parallel.

### Step 1 — Pre-Check (`01-pre-check-validation.sql`)

**Save the full output before proceeding.**

Captures baseline state: current index definitions, sizes, and DTU metrics. You will compare this output against Step 4 results.

- Duration: < 1 minute
- Risk: None (read-only)

### Step 2 — Drop Bloated Indexes (`02-drop-bloated-indexes.sql`)

Drops the 4 bloated indexes. Uses `ONLINE = ON` so existing queries are not blocked.

- Duration: ~15–30 minutes
- Risk: Low

Each `DROP` is isolated in its own `GO` batch and wrapped in an `IF EXISTS` check, making this script safe to re-run.

### Step 3 — Create Optimized Index (`03-create-optimized-index.sql`)

Builds the replacement index on 23.6M rows. Uses `ONLINE = ON` and `SORT_IN_TEMPDB = ON`.

- Duration: ~30–60 minutes
- Risk: Low — `ONLINE = ON` avoids table locks; temporary DTU spike expected during build

**Do not cancel this step once started.** A partial index build will be automatically rolled back by SQL Server if cancelled, but this wastes the build time.

### Step 4 — Post-Check Validation (`04-post-check-validation.sql`)

Confirms that the 4 bloated indexes are gone, the new index exists, and DTU metrics are improving.

- Duration: < 1 minute
- Risk: None (read-only)

**Compare output against the saved Step 1 baseline.**

## Rollback

**Script:** `05-rollback-recreate-indexes.sql`

### When to roll back

- Application errors tied to `DeviceSummaries` queries appear after Phase 1 completes
- Query performance is measurably worse after Phase 1 (compare execution plans)
- Explicit request from engineering or database lead

### When NOT to roll back

- DTU is still elevated immediately after — allow 30–60 minutes for metrics to stabilize
- Queries work correctly but response times are slightly different — investigate first

### How to roll back

1. Run `05-rollback-recreate-indexes.sql` during a maintenance window
2. Expected duration: **1.5–3 hours** (rebuilding all 4 indexes on 23.6M rows)
3. Verify using `01-pre-check-validation.sql` that the index state matches the original baseline

The rollback script:
- Drops the new `IX_DeviceSummaries_IMEI_TimeStamp_IsMove` index (if it exists)
- Recreates all 4 original indexes with their original column definitions
- Is wrapped in `IF NOT EXISTS` checks — safe to re-run

> **Note:** The rollback column lists for the 26-column and 35-column indexes were reconstructed from the schema at migration time. Before running the rollback, verify the column lists against the saved output of `01-pre-check-validation.sql`.
