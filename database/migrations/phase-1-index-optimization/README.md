# Phase 1: Index Optimization

## Purpose

Phase 1 covers index clean-up and query-pattern optimisation across the five
highest-impact tables in the database.  No data is changed, no schema columns
are altered, and no stored procedures are modified.  Only index definitions and
a new archival object for `GeocodeLocationLogs` are touched.

---

## Tables in Scope

| Table | Rows | Action |
|---|---|---|
| `DeviceSummaries` | 23.6M | Drop 3 bloated indexes → 1 narrow covering index |
| `DeviceEvents` | 19.4M | Drop bloated indexes (identified from pre-check) → 1 narrow covering index |
| `AdvanceTrackingSettingSummaries` | 18.6M | Drop bloated indexes (identified from pre-check) → 1 narrow covering index |
| `GeocodeLocationLogs` | 17.2M | Archival stored procedure + scheduled purge (no retention policy existed) |
| `TrackedAssets` | 46K | Add covering index to convert full-table scans to seeks |

### Step 1 — Pre-Check (`01-pre-check-validation.sql`)

**Save the full output before proceeding.**

Captures baseline state: current index definitions and sizes for all five
target tables, plus a DTU snapshot.

- Duration: < 1 minute
- Risk: None (read-only)

### Step 2 — Drop Bloated Indexes (`02-drop-bloated-indexes.sql`)

Drops the 3 known bloated indexes on `DeviceSummaries`.

For `DeviceEvents` and `AdvanceTrackingSettingSummaries`, the script contains
**commented-out placeholder blocks** — fill in the actual index names from the
Step 1 output before running.

- Duration: ~15–30 minutes
- Risk: Low — `ONLINE = ON` so existing queries are not blocked
- Safe to re-run: every `DROP` is wrapped in `IF EXISTS`

### Step 3 — Create Optimized Indexes (`03-create-optimized-indexes.sql`)

Builds four replacement indexes across `DeviceSummaries`, `DeviceEvents`,
`AdvanceTrackingSettingSummaries`, and `TrackedAssets`.  Uses `ONLINE = ON`
and `SORT_IN_TEMPDB = ON` throughout.

- Duration: ~2–6 hours total (ONLINE builds on large tables)
- Risk: Low — temporary DTU spike expected during each build
- **Do not cancel once started.** A partial build is automatically rolled back
  by SQL Server if cancelled, wasting the build time.

### Step 4 — GeocodeLocationLogs Archival (`04-geocodelocationlogs-archival.sql`)

Creates `GeocodeLocationLogs_Archive` and the stored procedure
`usp_ArchiveGeocodeLocationLogs`.

**ACTION REQUIRED:** Identify the date/timestamp column name from the
`GeocodeLocationLogs` table (check Step 1 output — common names:
`CreatedDate`, `LogDate`, `EventDate`, `Timestamp`, `CreatedAt`) and pass it
as `@DateColumnName` when calling the procedure.

After the archival objects are created, perform the first manual run:
```sql
EXEC usp_ArchiveGeocodeLocationLogs
    @RetentionDays  = 90,
    @DateColumnName = N'<actual_column>';
```

Then schedule a daily SQL Agent job using the same `EXEC`.

- Duration: 30–90 minutes (first run on 17.2M rows)
- Risk: Low — rows are moved to archive, not deleted

### Step 5 — Post-Check Validation (`05-post-check-validation.sql`)

Confirms the expected index state on all five tables, verifies the archival
ran, and checks that DTU metrics are trending downward.

- Duration: < 1 minute
- Risk: None (read-only)

**Compare all output against the saved Step 1 baseline.**

## Rollback

**Script:** `06-rollback.sql`

### When to roll back

- Application errors tied to any Phase 1 table appear after changes complete
- Query performance is measurably worse (compare execution plans)
- Explicit request from engineering or database lead

### When NOT to roll back

- DTU is still elevated immediately after — allow 30–60 minutes to stabilise
- Queries work correctly but response times differ slightly — investigate first

### How to roll back

1. Run `06-rollback.sql` during a maintenance window
2. Expected duration: **3–6 hours** (rebuilding all indexes across all tables)
3. Verify using `01-pre-check-validation.sql` that the index state matches the original baseline

The rollback script:
- Drops all four new optimized indexes (if they exist)
- Recreates the three known `DeviceSummaries` indexes with their exact original
  column definitions
- Contains template blocks for `DeviceEvents` and `AdvanceTrackingSettingSummaries`
  — fill in from the Step 1 pre-check output
- Does **not** drop the `GeocodeLocationLogs_Archive` table or the
  `usp_ArchiveGeocodeLocationLogs` procedure — archived rows can be
  re-inserted manually if needed (see comments in `06-rollback.sql`)
